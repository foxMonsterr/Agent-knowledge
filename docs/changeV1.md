# LearnAgent 需求升级说明书（changeV1）

> 版本：v1.0.1
> 日期：2026-06-01
> 目标：在现有 SmartAgent 代码基础上，将产品重心从“通用型 Agent Demo”升级为“面向个人学习的知识库 Agent”，并引入 ReAct 推理链路可视化、知识沉淀、学习闭环与 React 前端体验。

---

## 1. 文档背景

当前项目已经具备较完整的后端 Agent 能力、RAG 能力、记忆能力、流式对话能力，以及一个基础的前端管理台。但从产品定位上看，现阶段的能力更像“功能拼装的通用智能体平台”，用户很难快速理解这个 Agent 到底服务什么场景、为谁服务、为什么存在。

因此，本次改造建议将项目统一收敛为一个明确的方向：

**LearnAgent = 个人学习知识库 Agent + ReAct 推理可视化 + React 交互体验**

这个定位可以把现有能力重新组织成一个闭环：

- 输入学习资料
- 与 Agent 对话学习
- 记录推理过程
- 归纳为笔记
- 生成练习与闪卡
- 根据掌握情况复习
- 形成知识图谱
- 再反哺下一轮学习

---

## 2. 现状理解：基于代码与项目结构的真实约束

结合我对项目结构、后端源码与现有 PRD 的读取，当前项目已经有以下基础：

### 2.1 已存在的能力

- Spring Boot 后端主干已完成
- JWT 登录体系已完成
- 统一返回体、异常处理、审计链路、限流、追踪基本齐备
- 多模型配置已存在，具备主模型与兜底模型
- Chat Memory 已存在
- RAG 文档入库与向量检索已经具备
- SSE 流式输出已存在
- 前端是一个完整的 Vue 管理台，并不是纯空壳

### 2.2 当前产品问题

- 功能多，但主题分散，像平台，不像一个明确产品
- 现有的 Agent 类型偏“演示型”：工具调用、规划、知识库问答、全能对话都在，但缺少统一叙事
- 前端虽然完整，但页面和业务主线之间的关系不够聚焦
- 没有形成“学习-沉淀-练习-复习-分析”的闭环
- ReAct 只是一个架构方向，还没有成为用户可感知的核心体验

### 2.3 改造的核心原则

1. **一个核心主线**：所有能力围绕个人学习知识库展开
2. **一个核心交互**：ReAct 对话成为主入口
3. **一个核心资产**：个人知识笔记与学习记录
4. **一个核心反馈**：掌握度、复习提醒、薄弱点、知识图谱
5. **一个前端表达**：React 负责“推理链路与学习过程”的可视化呈现

## 3. 产品定义重构

### 3.1 新的产品定位

| 项目 | 说明 |
|------|------|
| 产品名称 | LearnAgent |
| 产品定位 | 面向个人学习者的 AI 知识库与学习助手 |
| 核心关键词 | 学习、知识沉淀、ReAct、复习、图谱、可视化 |
| 核心价值 | 让 AI 不只是回答问题，而是帮助用户把知识学会、记住、串联起来 |

### 3.2 产品一句话定义

**LearnAgent 是一个面向个人学习的 AI 知识库 Agent，它通过 ReAct 推理过程帮助用户学习、整理、练习和复习知识，并将知识不断沉淀为可检索、可追踪、可视化的个人学习资产。**

### 3.3 目标用户

| 角色 | 说明 |
|------|------|
| 学习者 | 学生、自学者、备考者、技术学习者、知识工作者 |
| 管理员 | 系统初始化账号，负责系统级管理与监控 |

### 3.4 核心使用场景

- 学习某个主题时，先问 Agent 再整理成笔记
- 对某个知识点不理解时，让 Agent 用 ReAct 方式一步步讲清楚
- 学完后让 Agent 自动生成闪卡、测验题
- 定期收到复习提醒，查看薄弱点
- 用知识图谱查看知识之间的联系
- 回放历史推理链路，知道自己是如何学会某个知识点的

## 4. 基于现有代码的重组建议

下面不是“完全推倒重来”，而是尽量对齐当前代码的可复用能力进行重新编排。

### 4.1 当前后端能力如何映射到 LearnAgent

| 现有能力 | 建议定位 |
|----------|----------|
| `ChatAgent` | 保留为基础聊天能力，可作为 LearnAgent 的通用问答底座 |
| `RagAgent` | 保留并升级为学习知识检索与引用来源能力 |
| `PlanningAgent` | 改造成“学习计划规划器”或“学习任务拆解器” |
| `FullAgent` | 建议重构为 LearnAgent 主入口，承载 ReAct 主流程 |
| `StreamAgent` | 建议并入 ReAct 流式输出链路，减少入口分裂 |
| `DocParseTool` | 建议升级为知识导入/笔记导入能力，而不是仅限 docs 目录读文件 |
| `DbQueryTool` | 建议替换为学习数据查询工具，而不是模拟员工部门查询 |
| `TranslateTool` | 若保留，应降级为辅助工具，不作为主线 |
| `DateTimeTool` | 可保留为学习提醒、复习时间计算辅助工具 |
| `CalculatorTool` | 可保留，作为学习中的数学/统计辅助工具 |

### 4.2 当前前端能力如何映射到 LearnAgent

| 现有页面 | 建议定位 |
|----------|----------|
| `stream` | 核心 ReAct 对话页，作为主入口 |
| `knowledge` | 知识库管理页，升级为“个人笔记/资料库” |
| `planning` | 学习计划页，聚焦计划生成与进度跟踪 |
| `session` | 学习会话与推理回放页 |
| `monitor` | 学习统计与系统监控合并页 |
| `home` | 学习仪表盘首页 |
| `agent` | 可并入 ReAct 对话页或学习工具中心 |

## 5. 模块化重构方案

建议将 LearnAgent 的能力重组为 8 个核心模块，而不是继续沿用偏平台化的功能命名。

```
LearnAgent
├── M1 ReAct 推理引擎
├── M2 个人知识库
├── M3 学习辅助工具
├── M4 学习计划与任务
├── M5 闪卡与测验
├── M6 学习记录与掌握度
├── M7 知识图谱与关联发现
└── M8 React 前端体验层
```

## 6. 模块一：ReAct 推理引擎

### 6.1 模块目标

ReAct 不应只是“日志式输出”，而要成为用户可理解、可介入、可回放的学习推理过程。

### 6.2 设计原则

1. 每次回答不是黑盒，而是 Thought → Action → Observation → Final Answer
2. 每个步骤都能在前端展示
3. 每次工具调用都能关联到学习目标
4. 每次推理结果都能沉淀为知识资产

### 6.3 建议增强点

#### 6.3.1 ReAct 状态拆分

建议将一次 ReAct 请求拆成以下状态：

- `idle`
- `thinking`
- `acting`
- `observing`
- `summarizing`
- `finalizing`
- `completed`
- `failed`

这样前端可以清楚知道当前步骤，而不是只看到流式文本。

#### 6.3.2 ReAct 步骤数据结构建议

| 字段 | 说明 |
|------|------|
| traceId | 整条推理链唯一标识 |
| conversationId | 所属会话 |
| stepNumber | 步骤编号 |
| stepType | thought / action / observation / final_answer |
| content | 当前步骤内容 |
| toolName | 工具名称 |
| toolInput | 工具参数 |
| toolOutput | 工具返回结果 |
| modelName | 当前使用模型 |
| timestamp | 时间戳 |
| latencyMs | 当前步骤耗时 |
| status | success / failed / interrupted |

#### 6.3.3 ReAct 对话的产品化定义

ReAct 对话不只是“问答”，而是：

- 解释概念
- 找资料
- 提炼笔记
- 生成例子
- 生成练习
- 引导复习
- 回答后自动沉淀学习成果

### 6.4 API 建议

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/learn/react/chat` | ReAct 同步对话 |
| GET | `/api/v1/learn/react/chat/stream` | ReAct 流式对话 |
| GET | `/api/v1/learn/react/traces/{traceId}` | 查看推理链路详情 |
| GET | `/api/v1/learn/react/sessions/{conversationId}` | 查看某会话的推理历史 |
| POST | `/api/v1/learn/react/intervene` | 用户中途补充信息 |
| POST | `/api/v1/learn/react/stop` | 中断当前推理 |

### 6.5 和 React 前端的契合点

前端可以通过 React 做到：

- 推理步骤卡片逐条展示
- 当前执行步骤高亮
- 工具调用结果与知识来源联动
- 可折叠思考链
- 可回放整个推理过程
- 允许用户在某一步插入补充说明

## 7. 模块二：个人知识库

### 7.1 模块目标

知识库不是单纯文档存储，而是个人学习资产中心。

### 7.2 数据建议

个人知识库建议分成三层：

1. **原始资料层**：上传的文档、网页摘录、笔记原文
2. **结构化笔记层**：提炼后的知识笔记、摘要、标签、分类
3. **检索向量层**：用于 RAG 检索与语义关联的片段向量

### 7.3 建议功能

| 功能 | 说明 |
|------|------|
| 笔记创建 | 手动创建学习笔记 |
| 笔记编辑 | 支持标题、正文、标签、分类、来源 |
| 笔记归档 | 学完的内容可归档，不删除但减少干扰 |
| 资料导入 | txt/md/pdf/网页内容导入 |
| 自动摘要 | 导入后自动生成摘要 |
| 自动标签 | 根据内容自动补标签 |
| 语义检索 | 按问题查找相关笔记 |
| 笔记引用 | 回复中显示所引用的笔记来源 |
| 笔记关联 | 自动发现相关笔记 |

### 7.4 建议新增的知识对象

| 对象 | 作用 |
|------|------|
| `KnowledgeNote` | 个人学习笔记主实体 |
| `KnowledgeSource` | 笔记来源信息 |
| `NoteChunk` | 向量切片 |
| `NoteRelation` | 笔记关系 |
| `NoteTag` | 标签体系 |
| `NoteCategory` | 分类体系 |

### 7.5 API 建议

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/learn/notes` | 创建笔记 |
| GET | `/api/v1/learn/notes` | 笔记列表 |
| GET | `/api/v1/learn/notes/{noteId}` | 笔记详情 |
| PUT | `/api/v1/learn/notes/{noteId}` | 更新笔记 |
| DELETE | `/api/v1/learn/notes/{noteId}` | 删除/归档笔记 |
| GET | `/api/v1/learn/notes/search` | 语义检索 |
| POST | `/api/v1/learn/notes/import` | 导入资料 |
| GET | `/api/v1/learn/notes/related/{noteId}` | 查看关联笔记 |

## 8. 模块三：学习辅助工具

### 8.1 模块目标

把 AI 的“讲解能力”延伸为“训练能力”。

### 8.2 工具建议

#### 8.2.1 知识工具

- 搜索笔记
- 创建笔记
- 更新笔记
- 查找关联笔记
- 获取笔记摘要

#### 8.2.2 练习工具

- 自动生成测验题
- 自动生成闪卡
- 自动评估答案
- 自动评估解释质量

#### 8.2.3 学习追踪工具

- 获取掌握度
- 获取复习提醒
- 记录学习行为
- 分析薄弱点

### 8.3 业务闭环建议

用户每次学习行为都应当形成以下闭环：

```
学习内容输入 → ReAct 讲解 → 笔记沉淀 → 生成测验/闪卡 → 复习 → 掌握度变化 → 推荐下一步学习
```

### 8.4 API 建议

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/learn/quiz/generate` | 生成测验 |
| POST | `/api/v1/learn/quiz/evaluate` | 评估答案 |
| POST | `/api/v1/learn/flashcards/generate` | 生成闪卡 |
| POST | `/api/v1/learn/flashcards/review` | 闪卡复习 |
| GET | `/api/v1/learn/study/progress` | 学习进度 |
| GET | `/api/v1/learn/study/weakness` | 薄弱点分析 |
| GET | `/api/v1/learn/study/reminders` | 复习提醒 |
| POST | `/api/v1/learn/study/record` | 手动记录学习行为 |

## 9. 模块四：学习计划与任务

### 9.1 模块目标

从“临时问答”升级为“学习过程管理”。

### 9.2 建议功能

- 根据主题自动拆解学习任务
- 将大目标拆为短期学习路径
- 给出学习顺序建议
- 对计划执行进度进行追踪
- 对低掌握度内容自动加入复习清单

### 9.3 计划建议结构

| 字段 | 说明 |

| planId | 计划唯一标识 |
| title | 计划名称 |
| topic | 学习主题 |
| goal | 学习目标 |
| steps | 学习步骤列表 |
| progress | 当前进度 |
| dueDate | 截止时间 |
| status | not_started / active / paused / completed |

### 9.4 API 建议

| 方法 | 路径 | 说明 |

## 10. 模块五：闪卡与测验

### 10.1 模块目标

把“理解了”变成“记住了”。

### 10.2 闪卡建议

- 正面：概念、问题、代码片段、关键词
- 背面：解释、答案、例子、对比
- 支持按标签与笔记批量生成
- 支持间隔重复提醒

### 10.3 测验建议

- 选择题
- 判断题
- 简答题
- 结合笔记自动出题
- 支持答案解释

### 10.4 闭环建议

测验结果与闪卡复习结果应反馈到掌握度模型中：

- 得分高 → 掌握度上升
- 连续错误 → 薄弱点提醒
- 多次遗忘 → 自动加入重点复习

## 11. 模块六：学习记录与掌握度

### 11.1 模块目标

记录用户的学习过程，并让系统真正“知道用户学会了什么”。

### 11.2 需要记录的行为

- 阅读笔记
- ReAct 对话
- 测验答题
- 闪卡复习
- 笔记编辑
- 资料导入
- 计划推进

### 11.3 建议增加的指标

| 指标 | 说明 |
|------|------|
| 今日学习时长 | 当天累计学习时间 |
| 本周学习次数 | 学习行为总次数 |
| 当前掌握度 | 主题或笔记的掌握评分 |
| 待复习数量 | 今日待复习项目数 |
| 薄弱主题数 | 掌握度低于阈值的主题数 |
| 连续学习天数 | 学习打卡连续天数 |

### 11.4 掌握度建议模型

掌握度不要只看一次测验，而应综合：

- 对话质量
- 测验得分
- 闪卡复习结果
- 主动编辑笔记次数
- 关联笔记浏览频率

这样会更像真实学习产品，而不是单次问答系统。

## 12. 模块七：知识图谱与关联发现

### 12.1 模块目标

让用户不仅知道“我学过什么”，还知道“知识之间是什么关系”。

### 12.2 节点与边的建议

| 类型 | 内容 |
|------|------|
| 节点 | 笔记、主题、课程、标签、概念 |
| 边 | 引用、相似、依赖、包含、对比 |

### 12.3 图谱能力建议

- 点击节点查看详情
- 从一个主题向外扩展相关知识
- 按掌握度高低着色
- 按标签显示不同簇
- 支持局部图谱与全局图谱切换

### 12.4 对前端 React 的意义

知识图谱是 React 非常适合展示的场景，因为它天然是复杂状态驱动的图结构视图。

可以体现：

- 节点状态管理
- 图数据联动过滤
- 选中/悬浮/拖拽交互
- 局部刷新与高性能渲染

## 13. 模块八：React 前端体验层

### 13.1 前端方向调整

原来的 Vue 管理台虽然完整，但如果产品主题转为“个人学习助手”，前端表达应更偏“学习操作台”，而不是“后台管理系统”。

如果坚持融入 React，建议至少在文档层面把 React 的价值明确下来：

- ReAct 推理链路可视化
- 学习流程卡片化
- 笔记与测验组件化
- 图谱状态联动
- 闪卡翻转交互

### 13.2 React 的角色定义

这里建议把 React 理解为两层含义：

1. **产品层 ReAct**：思考 + 行动 + 观察
2. **技术层 React 前端**：更适合做复杂交互和组件化表达

如果后续要真正迁移前端，可以将“学习操作台”重建为 React 应用；如果暂时不迁移，也可以先把页面交互设计为 React 友好的组件模式。

### 13.3 推荐的前端页面结构

| 页面 | 主要职责 |
|------|----------|
| 首页 | 学习概览、今日任务、待复习提醒 |
| ReAct 对话页 | 核心学习入口 |
| 知识库页 | 笔记与资料管理 |
| 测验页 | 生成与答题 |
| 闪卡页 | 复习与记忆训练 |
| 图谱页 | 知识关系可视化 |
| 统计页 | 学习趋势与薄弱点 |
| 设置页 | 模型、主题、提醒配置 |

## 14. 建议的接口分层

为了让后续模块更清晰，建议接口分层统一成以下风格：

- `/api/v1/auth/*`：认证
- `/api/v1/learn/react/*`：ReAct 对话与推理链
- `/api/v1/learn/notes/*`：知识笔记
- `/api/v1/learn/quiz/*`：测验
- `/api/v1/learn/flashcards/*`：闪卡
- `/api/v1/learn/study/*`：学习追踪
- `/api/v1/learn/graph/*`：知识图谱
- `/api/v1/admin/*`：系统管理
- `/api/v1/monitor/*`：系统监控

这样比当前“chat / agent / planning / stream / knowledge”横向拆法更符合学习产品的逻辑。

### 15.1 重命名建议

## 17. 最终建议

如果目标是让这个项目真正成为一个“有辨识度的 Agent 产品”，那么最重要的不是继续堆更多接口，而是把以下三件事做实：

1. **让用户知道它是学习助手**
2. **让用户看到 AI 是怎么思考的**
3. **让每一次对话都能沉淀为可复用知识**

如果这三点成立，LearnAgent 才真正从一个“空架子”变成一个“有学习闭环的个人知识库 Agent”。
